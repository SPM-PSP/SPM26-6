## 一、背景与目的

### 1.1 背景

本项目采用敏捷开发模式，边开发边测试。在实际开发过程中，可能存在以下变化因素：

- **需求变化**：用户需求调整、功能优先级变化
- **技术变化**：技术方案调整、第三方服务变更
- **人员变化**：成员时间冲突、任务进度偏差
- **环境变化**：政策要求、学校安排调整

### 1.2 目的

建立灵活的变更管理机制，确保：
- 变化可控、可追溯
- 项目进度不受重大影响
- 团队沟通顺畅
- 交付质量有保障

## 二、敏捷开发原则

### 2.1 核心原则

1. **个体和互动** 高于 流程和工具
2. **工作的软件** 高于 详尽的文档
3. **客户合作** 高于 合同谈判
4. **响应变化** 高于 遵循计划

### 2.2 本项目敏捷实践

| 实践 | 具体做法 |
|------|---------|
| **迭代开发** | 12 周分为 6 个迭代，每 2 周一个迭代 |
| **每日站会** | 微信群快速同步进度（可选） |
| **迭代评审** | 每迭代结束演示可工作的功能 |
| **迭代回顾** | 总结经验教训，持续改进 |
| **用户故事** | 功能以用户故事形式描述 |
| **持续集成** | 代码提交后自动构建测试 |

## 三、变更分类

### 3.1 变更类型

| 变更类型 | 说明 | 影响程度 | 审批流程 |
|---------|------|---------|---------|
| **微小变更** | 不影响整体进度，工作量变化<4 小时 | 低 | 任务负责人自行决定 |
| **一般变更** | 影响局部功能，工作量变化 4-16 小时 | 中 | 项目负责人审批 |
| **重大变更** | 影响核心功能或整体进度，工作量变化>16 小时 | 高 | 团队讨论 + 负责人审批 |

### 3.2 常见变更场景

#### 3.2.1 需求变更

| 场景 | 示例 | 应对措施 |
|------|------|---------|
| 功能新增 | 增加新的 AI 功能 | 评估优先级，可放入后续迭代 |
| 功能删减 | 某功能使用频率低 | 及时止损，减少开发工作量 |
| 功能调整 | 修改作业提交流程 | 评估影响范围，调整相关模块 |
| 优先级变化 | 核心功能优先 | 调整迭代计划，优先开发高优先级功能 |

#### 3.2.2 技术变更

| 场景 | 示例 | 应对措施 |
|------|------|---------|
| 技术方案调整 | 更换 AI 服务商 | 保持接口封装，降低切换成本 |
| 第三方服务变更 | API 涨价/限流 | 准备备选方案，快速切换 |
| 技术难点 | 某功能实现困难 | 团队攻关或简化实现 |

#### 3.2.3 人员变更

| 场景 | 示例 | 应对措施 |
|------|------|---------|
| 时间冲突 | 成员考试/活动 | 调整任务分配，其他成员协助 |
| 进度偏差 | 任务延期 | 及时调整后续任务，必要时加班 |
| 成员退出 | 特殊情况退出 | 知识共享，快速交接 |

## 四、变更管理流程

### 4.1 变更提出

任何团队成员均可提出变更，方式：
- 微信群提出
- GitHub Issue 提交
- 例会讨论

**变更提出模板**：
```
【变更申请】
提出人：XXX
日期：2026-XX-XX
变更类型：需求/技术/人员
变更内容：简要描述
变更原因：为什么需要变更
影响分析：对进度、工作量、其他功能的影响
建议方案：建议如何处理
```

### 4.2 变更评估

**项目负责人评估内容**：
- 变更必要性
- 影响范围（功能、进度、工作量）
- 资源需求
- 风险评估

**评估时限**：
- 微小变更：即时决定
- 一般变更：24 小时内决定
- 重大变更：48 小时内团队讨论决定

### 4.3 变更审批

| 变更类型 | 审批人 | 审批时限 |
|---------|--------|---------|
| 微小变更 | 任务负责人 | 即时 |
| 一般变更 | 项目负责人 | 24 小时 |
| 重大变更 | 团队讨论 + 负责人 | 48 小时 |

### 4.4 变更实施

**审批通过后**：
1. 更新项目计划（项目计划书、迭代计划）
2. 更新任务分配（GitHub Projects）
3. 通知相关成员
4. 实施变更
5. 跟踪变更效果

### 4.5 变更跟踪

**跟踪内容**：
- 变更实施进度
- 实际工作量 vs 预估工作量
- 对整体进度的影响
- 是否达到预期效果

**变更记录**：
所有变更记录在《变更管理日志》中（见附录 A）。

## 五、风险应对策略

### 5.1 进度风险

**风险**：任务延期，影响后续开发

**预警机制**：
- 任务进度<50% 且时间已过半 → 黄色预警
- 任务进度<30% 且时间已过 2/3 → 橙色预警
- 任务可能严重延期 → 红色预警

**应对措施**：
| 预警级别 | 应对措施 |
|---------|---------|
| 黄色预警 | 任务负责人加班赶工 |
| 橙色预警 | 其他成员协助，调整任务优先级 |
| 红色预警 | 项目负责人介入，重新分配任务，必要时削减功能 |

### 5.2 技术风险

**风险**：技术难点无法攻克、第三方服务不稳定

**应对措施**：
- **AI 服务不稳定**：多服务商备选，自动降级切换
- **技术难点**：团队攻关、查阅资料、寻求外部帮助
- **性能问题**：提前进行性能测试，优化关键路径

### 5.3 人员风险

**风险**：成员时间冲突、进度不一致

**应对措施**：
- **提前规划**：成员提前告知重要事件（考试、面试等）
- **知识共享**：代码文档化，避免单点依赖
- **任务备份**：关键任务有备选负责人
- **灵活调整**：根据成员时间动态调整任务

### 5.4 需求风险

**风险**：需求频繁变更、范围蔓延

**应对措施**：
- **需求冻结**：迭代开始后，本迭代需求不变更
- **优先级管理**：MoSCoW 法则（Must/Should/Could/Won't）
- **最小可行产品（MVP）**：优先保证核心功能
- **变更审批**：重大变更需团队讨论

## 六、迭代计划调整

### 6.1 迭代周期

原计划 12 周，每 2 周一个迭代，共 6 个迭代：

| 迭代 | 周次 | 主题 |
|------|------|------|
| Iteration 1 | 第 1-2 周 | 需求分析与系统设计 |
| Iteration 2 | 第 3-4 周 | 基础框架 + 用户认证 |
| Iteration 3 | 第 5-6 周 | 核心功能开发（一） |
| Iteration 4 | 第 7-8 周 | 核心功能开发（二） |
| Iteration 5 | 第 9-10 周 | 功能完善 + 测试优化 |
| Iteration 6 | 第 11-12 周 | 集成测试 + 部署交付 |

### 6.2 迭代调整规则

**调整时机**：
- 迭代评审后发现进度偏差
- 重大变更获批后
- 外部因素导致计划变更

**调整方式**：
1. **功能调整**：增减功能、调整优先级
2. **时间调整**：延长/缩短迭代周期（慎用）
3. **资源调整**：增加/减少人力投入
4. **范围调整**：削减非核心功能

### 6.3 迭代调整流程

```
迭代评审 → 识别偏差 → 分析原因 → 
制定调整方案 → 团队讨论 → 负责人审批 → 
更新计划 → 通知成员 → 执行新计划
```

## 七、沟通机制

### 7.1 日常沟通

| 方式 | 频率 | 参与人 | 内容 |
|------|------|--------|------|
| 微信群 | 随时 | 全体 | 日常沟通、问题讨论 |
| GitHub Issues | 随时 | 全体 | 问题跟踪、任务讨论 |
| 站会（可选） | 每日 | 全体 | 快速同步进度（5-10 分钟） |

### 7.2 定期会议

| 会议 | 频率 | 参与人 | 时长 | 内容 |
|------|------|--------|------|------|
| 迭代计划会 | 每迭代开始 | 全体 | 1-2h | 制定迭代计划、任务分配 |
| 迭代评审会 | 每迭代结束 | 全体 | 1h | 演示功能、收集反馈 |
| 迭代回顾会 | 每迭代结束 | 全体 | 1h | 总结经验、持续改进 |
| 项目例会 | 每周 | 全体 | 30min | 进度同步、问题讨论 |

### 7.3 变更沟通

**变更提出后**：
1. 微信群/Issues 通知相关成员
2. 一般变更：负责人决定后通知
3. 重大变更：召开临时会议讨论
4. 变更结果：更新计划并通知全体

## 十、总结

### 10.1 核心要点

1. **拥抱变化**：变化是正常的，关键是如何管理
2. **快速响应**：小步快跑，快速迭代
3. **透明沟通**：信息同步，避免信息差
4. **质量优先**：进度再紧也要保证质量
5. **团队至上**：相互支持，共同进退

### 10.2 成功标准

- 按时交付可用的软件
- 核心功能全部实现
- 代码质量达标
- 团队成员满意
- 用户反馈良好

### 附录 D：常见问题解答

**Q1：变更太多怎么办？**
A：区分优先级，优先保证核心功能（Must Have），非核心功能可延后或削减。

**Q2：成员进度严重滞后怎么办？**
A：及时预警，其他成员协助，必要时调整任务分配或加班赶工。

**Q3：技术难点无法攻克怎么办？**
A：团队攻关、查阅资料、寻求外部帮助，或简化实现方案。

**Q4：需求频繁变更如何控制？**
A：迭代开始后需求冻结，重大变更需团队讨论审批，使用 MoSCoW 法则管理优先级。

**Q5：如何平衡进度和质量？**
A：质量是底线，进度可调整。通过持续集成、代码审查、自动化测试保证质量。
