AI 学伴 - 项目测试计划书 (V1.0)
文档状态：[正式]编写人：张炜绅（测试工程师）审核人：陈城（项目负责人）编写日期：项目第 1 周
1. 概述
1.1 项目背景
“AI 学伴” 是一个面向高校师生和小团队的轻量化 AI 辅助学习平台。本项目旨在通过集成第三方大语言模型（如文心一言、通义千问等），解决高校师生在答疑、作业批改、试题生成等场景下的效率痛点。
1.2 测试目的
本计划书旨在为 “AI 学伴” 项目的测试工作提供全局指导，明确测试范围、测试策略、资源安排及进度规划。确保平台在第 12 周交付时，功能完整、性能达标、系统稳定，且核心业务流程（如 AI 交互）无严重缺陷。
2. 测试范围
2.1 包含范围 (In-Scope)
前端测试 (Web 端)：学生端、教师端、管理员端的 UI 界面展示、交互逻辑、路由跳转及跨浏览器兼容性。
后端 API 测试：核心业务逻辑接口、文件上传 / 下载接口、消息通知接口的正确性与安全性。
AI 功能集成测试：AI 答疑、AI 笔记总结、AI 错题分析、AI 批改作业、AI 出题模块的数据流转与容错处理。
系统级测试：用户角色权限控制、系统并发性能（重点测试外部 AI API 调用时的系统表现）。
2.2 不包含范围 (Out-of-Scope)
第三方 AI 大模型的底层算法准确度与内容合规性（依赖第三方服务商的保障）。
移动端原生 App 测试（本项目聚焦 Web 端多端适配）。

4. 测试环境需求
服务器环境：阿里云 / 腾讯云 2 核 4G 云服务器（划分为测试专区）。
数据库环境：MySQL 云服务（建立独立的 ai_companion_test 数据库，与开发数据物理隔离）。
客户端环境：Windows/macOS 系统，主流浏览器（Chrome 100+, Edge, Safari）。
代码与文档管理：使用 GitHub 进行代码版本控制与 GitHub Actions 辅助持续集成，Bug 统一提交至 GitHub Issues。
5. 测试进度安排
测试进度将严格对齐项目 12 周的总进度计划：
第 1 周：完成本《测试计划文档》的制定与评审。
第 2-3 周：编写核心模块的测试用例文档，并搭建自动化测试与接口测试环境。
第 4-7 周：核心迭代测试阶段。跟随开发进度，对用户管理、AI 答疑、AI 批改、教案 / 试题管理等四个模块进行单元测试与接口功能测试，提交第一轮 Bug。
第 8-9 周：跟进缺陷修复，对前端交互优化和新增的文件 / 通知功能进行测试。
第 10 周：系统集成测试阶段。开展全业务流程的集成测试、性能测试与初步安全测试。
第 11 周：回归测试阶段。进行第二轮 Bug 回归验证，协助进行 UI/UX 最终验收。
第 12 周：生产环境验收测试，输出最终《测试报告》，项目交付。
6. 缺陷管理与交付标准
6.1 缺陷等级定义
   
通过 GitHub Issues 追踪 Bug，并打上对应的标签（Label）：
🔴 Blocker (阻碍)：系统崩溃、核心 AI 接口彻底不通、无法登录。（必须当日修复）
🟠 Critical (严重)：核心功能流程中断，如无法提交作业、权限越界。
🟡 Major (一般)：功能逻辑错误，但有替代方案或不影响核心流程。
🔵 Minor (轻微)：UI 样式错位、文案拼写错误、非常规操作下的轻微异常。
6.2 测试通过标准
核心业务用例（P0/P1 级别）100% 执行并全部通过。
Blocker 和 Critical 级别的 Bug 修复率达到 100%。
Major 级别的遗留 Bug 数量不超过 3 个，且已在项目会议中评估接受风险。
完成所有计划内的性能与安全测试，且各项指标达到预期标准
